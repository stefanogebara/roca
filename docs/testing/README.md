## Running the suites

```bash
npm install
npm run typecheck   # tsc --noEmit  (the type gate)
npm test            # vitest run    (all suites once)
npm run test:watch  # vitest        (watch mode)
```

Vitest is configured in `vitest.config.ts`: `environment: 'node'`, tests matched by
`tests/**/*.test.ts`. Every suite is pure and deterministic — no network, no DB, no
API keys — so `npm test` runs offline and fast. There is **no committed CI workflow**
(no `.github/`); the suites are run locally/manually, with `typecheck` as the
compile gate.

## What each suite covers

| Suite | Module under test | Focus |
|-------|-------------------|-------|
| `tests/deltaT.test.ts` | `api/_lib/tools/deltaT.ts` | `wetBulbC` (Stull formula: saturation ≈ dry-bulb, hot-dry depression, a known reference point), `deltaT`, `assessHour` (go/caution/no-go across Delta T, wind, rain, and worst-condition-wins), and `sprayWindow` (now-good, next-good-hour, none-found, empty-throws). |
| `tests/compliance.test.ts` | `api/_lib/compliance.ts` | `checkOutbound`: passes an informational disease answer and a "registro existe" mention; **blocks** apply+`0,5 L/ha` and pulverize+`2 kg/ha`; allows calagem advice with a number (not a pesticide prescription); doesn't flag a bare area number. |
| `tests/calendar.test.ts` | `api/_lib/tools/calendar.ts` | `vazioStatus` (MT active on 2026-07-07, GO inclusive boundaries, out-of-window still informs, PR regional hedge, unknown/null UF stays silent, main soy states covered), `upcomingTransitions` (7-day window, proximity sort), `isCalendarStale` (fresh in-season, stale past it). |
| `tests/agrofit.test.ts` | `api/_lib/tools/agrofit.ts` | `normalizeCrop` (crop words → keys; non-focus → null), `lookupPest` (soy "ferrugem" → *Phakopsora* with registered products; accent- and scientific-name matching; sub-floor → null; all-crop search), `groundingBlock` (states what's registered, reasserts the receituário boundary, and **emits no dose**). |
| `tests/twilio-signature.test.ts` | `api/_lib/transport/twilio.ts` | `computeTwilioSignature` matches a reference vector generated by Twilio's own SDK (pins byte-equivalence so the SDK-free implementation can't silently diverge), is order-independent, and changes when any value changes. |
| `tests/cloud.test.ts` | `api/_lib/transport/cloud.ts` | `verifyCloudChallenge` (echo on token match, null on mismatch); `verifySignature` (accepts a correct `X-Hub-Signature-256`, rejects a tampered body and a missing header); `parseInbound` for text/image(caption)/voice/location, and `null` for status callbacks and malformed JSON. |
| `tests/crops.test.ts` | `api/_lib/tools/crops.ts` | `parseCrops` (single, multiple in stable order, pasture/cattle vocabulary, accent-insensitive café/citros, dedupe, empty when unrecognized, no false substring match) and `joinCrops` natural-language joins. |

## What is deliberately not unit-tested

The tests target pure, high-consequence logic (agronomic verdicts, the compliance
boundary, grounding, signature algorithms, parsing). The following are intentionally
left to live verification instead:

- **I/O tool wrappers** — `weather.ts` (Open-Meteo), `soil.ts` (SoilGrids), `geo.ts`
  (BigDataCloud): thin fetch adapters with fail-soft behaviour; exercised live.
- **LLM calls** — `llm.ts`, `router.ts`, `reason.ts`, `transcribe.ts`: non-
  deterministic and network-bound; validated end-to-end.
- **Persistence** — `db.ts`: verified against a real Supabase instance.
- **The webhook handler** — `webhook.ts`: verified with the harness below.

## The `simulate-inbound.mjs` live harness

`scripts/simulate-inbound.mjs` posts a **correctly-signed Twilio** request to the
deployed webhook (`https://roca-black.vercel.app/api/webhook`) so you can exercise the
whole production loop without a phone. It reads `TWILIO_AUTH_TOKEN` from `.env`, builds
the HMAC-SHA1 signature exactly as Twilio does, and prints the response status and
latency.

```bash
# text (defaults to "posso pulverizar hoje?")
node scripts/simulate-inbound.mjs "que praga é essa na minha soja?"

# location pin → farm card (lat,lon)
node scripts/simulate-inbound.mjs --location=-12.6,-55.7

# media (image → vision path); set the content type explicitly
node scripts/simulate-inbound.mjs --media-url=https://example.com/leaf.jpg --media-type=image/jpeg

# negative test: a wrong signature must be rejected with 403
node scripts/simulate-inbound.mjs --bad-signature "isso não deve passar"
```

Notes:

- The reply is delivered out-of-band (the webhook acks with empty TwiML), so the
  harness shows the ack, not the farmer-facing text. Confirm the actual reply on the
  WhatsApp thread, or inspect the `messages` table in Supabase.
- Because it sends a real signed request, it also exercises signature verification,
  adapter selection, the pipeline, LLM calls, and persistence — a genuine end-to-end
  check.

## Key/credential check scripts

- `scripts/check-openrouter.mjs <env-file>` — validates `OPENROUTER_API_KEY` (the key
  the app actually uses): checks key/credit state, runs a tiny live Haiku completion,
  and lists current `anthropic/*` slugs so you can confirm the model tiers resolve.
- `scripts/check-key.mjs <env-file> [VAR]` — validates an `ANTHROPIC_API_KEY` directly
  against the Anthropic API. This is a **legacy** diagnostic from before the OpenRouter
  migration; the app does not use `ANTHROPIC_API_KEY`. Prefer `check-openrouter.mjs`.

## Verification philosophy

- **Unit-test the load-bearing pure logic.** Anything where a wrong answer is
  expensive — a spray verdict, the prescription gate, Agrofit grounding, the vazio
  calendar, signature math, crop parsing — is deterministic and pinned by tests. The
  compliance and grounding tests are the "receipts" for the two design constraints
  (never prescribe, never invent).
- **Pin equivalence to external algorithms.** The Twilio signature test asserts
  byte-equality with the official SDK's output, and the Cloud test builds its own
  HMAC-SHA256 to check the verifier — so an edit can't silently diverge from a
  provider's scheme.
- **Verify the wired system live.** Network tools, LLM behaviour, and persistence are
  validated against the real deployment: send a signed request with
  `simulate-inbound.mjs`, confirm the reply and that rows land in Supabase, and check
  that the daily cron writes a `monitor_runs` row. See
  [the deployment doc](../deployment/) for the deploy/verify runbook and
  [the webhook doc](../webhook/) for the contract these tests defend.
